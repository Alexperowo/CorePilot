**Твой автономный цикл на каждую задачу:**

страховка: даже если история сессии недоступна, журнал восстановит контекст. Каждую
сессию начинай с его прочтения.

**Решения принимай сам, но безопасно.** Перед удалением/перезаписью больших кусков —
убедись (через запуск тестов), что не ломаешь. Спорные архитектурные развилки можно

## 2. ЧТО ТАКОЕ COREPILOT

Локальная мультиагентная «фабрика ИИ» для Windows. Цель — дать НЕтехническому
пользователю команду ИИ-агентов, работающую на его железе бесплатно (локальные модели)
или через бесплатные облачные API.

**Среда (целевая машина Александра):**
- Windows 10, чистая (без антивируса-хлама, без OneDrive). В простое ~2 ГБ RAM.
- Ryzen 3600, 16 ГБ RAM, **RX 6600 8 ГБ VRAM** (это потолок — помни про OOM), SSD, 4K монитор.
- Python 3.12.10. Строго БЕЗ WSL2 и Docker.

**Зависимости (сверены, не ломай совместимость):**
- `crewai==1.14.6` + `crewai[google-genai]==1.14.6` (Gemini у CrewAI — нативный провайдер)
- `litellm>=1.86.0,!=1.82.7,!=1.82.8` (CrewAI сам тянет совместимую; 1.82.7/8 скомпрометированы)
- `dearpygui>=2.0.0,<3.0.0` (десктоп UI), `pydantic>=2.7.0,<3.0.0`
- Мобильное: `flet==0.80.5` (ПИНОВАТЬ точно — символ `<` в версии ломает CMD-парсер при сборке APK)

**Три рабочих режима:**
1. **Конвейер** (Pipeline) — правка кода: gather → architect → fix → audit. End-to-end.
2. **Менеджер** (Manager) — DAG-бэклог задач + фоновый **Демон**, исполняющий их по
   зависимостям. Цель — оставить на ночь, утром получить результат.
3. **AI Cleaner** — очистка системы (реестр, мусорные файлы; Windows-specific).

**5 ролей агентов** (`ROLES = ["gatherer","architect","coder","auditor","oracle"]`):
Сборщик / Архитектор / Кодер / Аудитор / Оракул. У каждой — свой источник (бэкенд) и
модель. Разные модели на роль снижают галлюцинации.

## 3. АРХИТЕКТУРА И ПРИНЦИПЫ (НЕ НАРУШАТЬ)

- **Изоляция слоёв.** UI (`ui_*.py`) общается ТОЛЬКО с `service_layer.py`. UI НЕ
  импортирует `agents`, `daemon`, бизнес-логику напрямую. Это железно.
- **Zero Trust к выводу LLM.** Любой ответ модели парси/валидируй, не верь формату.
- **ASCII-маркеры** в коде/логах вместо эмодзи там, где идёт в файлы/консоль Windows
  (кодировки). В UI и чате с пользователем эмодзи допустимы.
- **Доступность.** Александр — слабовидящий. Крупный шрифт, высокий контраст в UI.
- **Без хардкода путей.** Пути привязывай к расположению скрипта (`__file__`), не к
  текущей рабочей папке. См. урок про AUTO_DIR ниже.

**Карта модулей (десктоп, 22 файла):**
- `app.py` — точка входа десктопа (запускает UI).

- `agents.py` — построение LLM, ключи (ротация), `safe_kickoff` (отказоустойчивый запуск
  crew с fallback), провайдеры. ~900 строк.

- `cleaner_flow.py` / `cleaner_agents.py` / `cleaner_tools.py` — AI Cleaner.

## 5. ЖУРНАЛ РЕШЁННЫХ БАГОВ (НЕ ревизовать — это выстрадано)

**Профили (самый упорный, ~6 итераций):**
- Корень №1: combo DearPyGui с ФИКСИРОВАННЫМИ строковыми тегами при пересоздании не
  оживают (зомби-алиасы освобождаются в конце кадра). РЕШЕНИЕ: combo ролей создаются с
  `dpg.generate_uuid()` (уникальные int-ID), при применении профиля контейнер
  `role_rows_container` чистится через `delete_item(children_only=True)` и строится заново.
- Корень №2 (настоящий, нашёлся в iter29): **демон в `process_tasks` сканировал все
  `*.json` в `auto_tasks` БЕЗ фильтра** → принимал `config_profiles.json` за задачу →
  переименовывал в `.processing` → профили «исчезали», но настройки (в другом файле)
  сохранялись. РЕШЕНИЕ: профили вынесены в `user_data/`; демон фильтрует `_is_system_file`;
  атомарная запись + восстановление застрявших `.processing` при чтении.
- Это НЕ был антивирус (у Александра чистая Windows) — гипотеза была неверной.

**Отказоустойчивость / провайдеры:**
- `_key_lock = threading.RLock()` (был дедлок: next_api_key брал лок и внутри звал
  init, тот же лок → облачные списки висли 20с).
- Межпровайдерный fallback в `safe_kickoff`: ротация ключей упавшего провайдера →
  резервный провайдер (`_next_backup_provider`, порядок cerebras→groq→gemini→
  openrouter→...) → локальная модель. Спасает ночной прогон демона.
- `peek_api_key` (чтение списков моделей НЕ двигает счётчик ротации).
- OpenRouter `auto` всегда первым в списке моделей.
- Обход бага CrewAI #5886: `install_cache_breakpoint_fix` чистит `cache_breakpoint` из
  сообщений для всех провайдеров кроме anthropic (Groq и др. его отвергают).
- 402/credits/insufficient добавлены в триггеры fallback.
- `response_format=json_object` ВЫКЛ по умолчанию для локальных моделей (LM Studio их
  отвергает, ломая первый запрос демона). Персистентный кэш `_NO_RESPONSE_FORMAT`.
- `_role_mode`: явный локальный бэкенд побеждает устаревшее `mode_=cloud` в сессии.

**Память / llama.cpp:**
- mmap: наш llama-server запускается с `--no-mmap` (иначе копия модели в RAM при весах
  на GPU → дублирование на 16 ГБ системе → своп). Дефолт контекста 4096 (не 8192) для
  8 ГБ VRAM. LM Studio: документировано снять «Use mmap» в TROUBLESHOOTING.md.
- Определение VRAM на Windows занижается (WMI отдаёт 32-битное число; для RX 6600 8 ГБ
  покажет ~4 ГБ) — есть ручной ввод объёма.
- Ollama порт 11434 (не 1234 — это LM Studio).

**Пути:**
- `AUTO_DIR` привязан к `os.path.dirname(os.path.abspath(__file__))`, НЕ к cwd (иначе при
  запуске из разных папок — разные `auto_tasks`, данные «теряются»). То же в daemon.py.

**Thread-safety (DearPyGui):**
- DPG НЕ потокобезопасен для мутаций графа (add_*/configure_item/delete_item). Прогресс из
  фоновых потоков идёт через очередь (`BackgroundRunner.push_progress/drain_progress`),
  применяется в ГЛАВНОМ потоке в render-цикле. Иначе SegFault.
- ОТКРЫТО (не решено): таймаут этапа в daemon через ThreadPoolExecutor НЕ убивает
  зависший поток (Python не умеет убивать треды) — модель висит в VRAM. Полное решение —
  multiprocessing-воркеры с terminate(). Отложено (крупный рефакторинг).

**Мобильное (APK):**
- Серый экран на старте: `load_settings` читал публичную папку (Download) в `__init__` до
  запроса прав → PermissionError → молчаливое падение. РЕШЕНИЕ: настройки в
  `client_storage`; crash logger в `main()` (traceback на экран + в storage); ensure_dirs
  обёрнут. ⚠️ ПРОВЕРИТЬ на устройстве, что crash logger реально показывает ошибку.
- Сборка: build_apk.bat авто-принимает лицензии SDK, находит Python/Git, добавляет
  Flutter в PATH динамически; flet запинен точной версией (символ `<` ломал CMD).
- **⚠️ АКТИВНЫЙ БАГ (приоритет — Александр подтвердил в последней сессии): APK ВСЁ ЕЩЁ
  не собирается, ругается на лицензии Android SDK**, несмотря на правку build_apk.bat
  (`echo y | sdkmanager --licenses`). Гипотезы для проверки ВЖИВУЮ на машине:
  (а) `sdkmanager.bat` лежит не в том пути, что перебирает батник — найди реальный путь
  через `where sdkmanager` или поиском в `%LOCALAPPDATA%\Android` и
  `%LOCALAPPDATA%\flet`; (б) лицензии принимает Flutter, а не sdkmanager — тогда нужен
  `flutter doctor --android-licenses` с автоподачей `y`, либо предварительно создать
  файлы-лицензии в `licenses/`; (в) Flet качает свой Android SDK в отдельную папку, и
  принимать лицензии надо именно там ПОСЛЕ первого `flet build`. ДЕЙСТВУЙ: запусти сборку
  на реальной машине, поймай точный текст ошибки (какие именно лицензии, какой путь),
  и почини по факту — у тебя есть зрение, не гадай. Это первейшая задача по мобильному.
