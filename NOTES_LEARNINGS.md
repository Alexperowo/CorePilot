# CorePilot — зафиксированные выводы (рабочая память проекта)

Сюда складываем проверенные практикой выводы, чтобы не наступать на грабли дважды.
Это НЕ пользовательская документация (та — в README/INSTALL и т.п.), а наши заметки.

---

## Локальные модели (LM Studio / llama.cpp)

- **`response_format={"type":"json_object"}` поддерживают не все локальные модели.**
  LM Studio-модели часто отвечают `does not support response_format`. Решение:
  capability-кэш `_NO_RESPONSE_FORMAT` в `agents.py` — связка (backend, model),
  один раз отвергнувшая параметр, больше его не получает. Парсер `_extract_json` —
  фолбэк. Не полагаться на принудительный JSON у локальных моделей.

- **«Model unloaded» — гонка выгрузки и JIT-загрузки.** Возникает, когда модель
  выгружена (нашим тумблером ИЛИ Auto-Evict LM Studio), а запрос пришёл раньше, чем
  сервер успел поднять её обратно. Решение: readiness-probe `_wait_local_model_ready`
  + «model unloaded» в триггерах повтора `safe_kickoff`. Тумблер «Выгружать между
  агентами» теперь после выгрузки ждёт готовности следующей модели.

- **VRAM 8 ГБ:** две модели 7-9B одновременно НЕ помещаются. Поэтому либо выгрузка
  между ролями (наш тумблер + probe), либо одна модель на все локальные роли.

- **Источник истины для VRAM:** не должно быть двух одновременно агрессивных
  механизмов. Наш тумблер теперь безопасен (с probe); Auto-Evict LM Studio может
  работать параллельно — probe сглаживает гонку.

## Имена ключей в secrets.toml

- Пользователи пишут имена ключей по-разному: `OPENROUTER_API_KEYS`, `OPENROUTER_KEYS`,
  `OPENROUTER_API_KEY`, `GOOGLE_API_KEY` (для gemini) и т.п. В `init_api_keys` —
  карта синонимов `_ALIASES` + умный фолбэк (любой ключ, начинающийся с имени
  провайдера). Не привязываться к одному написанию.

## OpenRouter

- Для списка моделей ключ НЕ нужен (публичный `/models`), для запросов — нужен.
- Конкретные `:free` модели часто НЕ поддерживают tool use (`No endpoints found
  that support tool use`). Для агентов с инструментами надёжнее `openrouter/auto`.

## Профили настроек (UI / DearPyGui)

- **`dpg.set_value` на комбо НЕ применит значение, которого нет в списке `items`.**
  Поэтому при применении профиля / перечитывании надо ПЕРЕСОБИРАТЬ items комбо
  (`configure_item(items=..., default_value=...)`), а не только `set_value`.
  Это была причина «профиль сохраняется, но не применяется».

- **Старое поле `mode_<role>` мешает новому `backend_<role>`.** Миграция
  `_migrate_role_sources` в `load_settings` доводит старые профили до нового формата.
  При сохранении `_collect` синхронизирует `mode_` с выбранным `backend_`.

## Облако / провайдеры

- `gemini/groq` не отдают остаток квоты через API — `quota=unknown` это НЕ баг.
  Только OpenRouter отдаёт остаток.
- Gemini free-tier: лимит ~20 запросов, частые 503 «high demand». Медленно и
  капризно — природа бесплатного тарифа. Ротация ключей обходит 429, но не ускоряет
  один запрос (этапы Конвейера последовательны).

## Дедлоки / потоки

- `_key_lock` должен быть `RLock` (реентерабельный): `next_api_key` внутри зовёт
  `_lazy_reload_keys`→`init_api_keys`, которые берут тот же лок. С обычным `Lock`
  был дедлок — облачные списки висли 20с.

## Установка / движок

- `setup.bat` и `Portable.bat` ставят llama.cpp ПО УМОЛЧАНИЮ (выбор Vulkan/CUDA/CPU).
  Философия: простой человек не обязан знать про LM Studio — движок «из коробки».
- `update_llama.bat` обновляет llama.cpp (backup старой версии). Находит и стационар,
  и портатив.

## Не-баги (не «чинить»)

- Вердикт «ОТКЛОНЕНО» Аудитора — легитимное решение агента, не сбой.
- `botocore` warning — litellm ищет AWS Bedrock, безвредно.

## Отказоустойчивость: вторая/третья линия обороны (iter22)
- **Межпровайдерный fallback** в safe_kickoff: если основной провайдер роли упал
  (429/quota/503/timeout/connection) И ротация его ключей не спасла — система пробует
  РЕЗЕРВНОГО провайдера с ключом (_next_backup_provider). Порядок предпочтения:
  cerebras, groq, gemini, openrouter, sambanova, mistral, huggingface, cohere,
  deepseek, openai, anthropic. Каждый резерв пробуется 1 раз за прогон (_tried_providers).
- Цепочка обороны: ротация ключей провайдера -> резервный провайдер(ы) -> локальная
  модель. Спасает НОЧНОЙ прогон Демона (демон идёт через safe_kickoff, fallback
  достаётся автоматически).
- **Квоты** работают для ВСЕХ провайдеров: OpenRouter через /auth/key (API), остальные
  через заголовки x-ratelimit-* реальных ответов (record_ratelimit_headers). До первого
  запроса честно 'unknown' (не выдумываем). Cerebras/Groq/др. шлют заголовки.

## Ревью Gemini (iter24) — внедрённые фиксы
- **Профили НЕ применялись — корень найден Gemini:** combo с ФИКСИРОВАННЫМИ строковыми
  тегами при пересоздании в DPG 2.x не оживают (старый алиас в очереди на освобождение
  до конца кадра -> «зомби-виджет», set_value стреляет в пустоту). РЕШЕНИЕ: combo ролей
  создаются с dpg.generate_uuid() (уникальные int-ID), при пересоздании просто чистим
  контейнер. AUTO_DIR-фикс был верным, но к этому багу отношения НЕ имел.
- **Thread-safety (риск SegFault):** _progress в PipelineTab звал dpg.configure_item из
  ФОНОВОГО потока. DPG не потокобезопасен для мутаций графа. РЕШЕНИЕ: BackgroundRunner
  получил очередь push_progress/drain_progress; события применяет ГЛАВНЫЙ поток в _tick.
- **Лог демона:** RotatingFileHandler (2МБ x1 backup) вместо усечения-при-старте —
  не забьёт диск при работе неделями.
- **peek_api_key:** чтение списка моделей больше не двигает счётчик ротации ключей
  (раньше открытие Настроек зря «прокручивало» ключи). list_cloud_models юзает peek.
- ОТКРЫТО (Gemini отметила, не критично сейчас): таймаут этапа в daemon через
  ThreadPoolExecutor НЕ убивает зависший поток (Python не умеет убивать треды) —
  модель может висеть в VRAM. Полное решение — multiprocessing-воркеры (отложено).

## ИСТИННАЯ причина исчезновения профилей (iter29) — НЕ антивирус, а наш демон!
СИМПТОМ: файл config_profiles.json.processing существует (с суффиксом .processing),
но в интерфейсе профилей нет; настройки при этом сохраняются.
КОРЕНЬ: профили хранились в auto_tasks/config_profiles.json. Демон в process_tasks
(daemon.py) брал ВСЕ *.json из AUTO_DIR БЕЗ фильтра системных файлов -> принимал
config_profiles.json за ЗАДАЧУ -> переименовывал в .processing (claim) -> код профилей
ищет config_profiles.json, не находит -> «0 профилей». Настройки сохранялись, т.к. они
в ОТДЕЛЬНОМ файле .ai_session.json (его демон не трогал по имени .json без точки... но
config_profiles попадал). service_layer._collect_task_statuses фильтровал по
_NON_TASK_FILES, а вот daemon.process_tasks — НЕТ. Вот почему «то есть, то нет».
ФИКС (тройная защита):
1. Профили перенесены в ОТДЕЛЬНУЮ папку user_data/ (вне auto_tasks) — демон туда не
   смотрит вообще.
2. daemon.process_tasks теперь фильтрует _is_system_file (config_profiles*, qa_history,
   .ai_session, no_response_format, quota_cache; учитывает суффиксы .processing/.tmp).
3. Атомарная запись профилей (tmp + os.fsync + os.replace) + восстановление застрявших
   .processing/.tmp при чтении (вернёт уже потерянные профили Local/Cloud).
+ Экспорт/импорт профилей в JSON (ручной бэкап, идея пользователя).
ВАЖНО: у пользователя чистая Windows без антивируса/OneDrive — подтвердило, что .processing
создаёт НАШ код (демон), а не среда. Гипотеза про антивирус была НЕВЕРНОЙ.

## Автоматизация установки движка llama.cpp для Android (09.06.2026)
ПРОБЛЕМА: движок llama-server для Android (arm64) должен быть ВСТРОЕН в APK
(nativeLibraryDir — единственное исполняемое место на нерутованном Android), но
скрипты download_engine.bat скачивали .so файлы и выводили инструкции для РУЧНОГО
добавления секции [tool.flet.android.libs] в pyproject.toml. Это было узким местом.

РЕШЕНИЕ — 3 новых скрипта:
- `_download_engine.ps1` — PowerShell: скачивает последний android-arm64 релиз с GitHub
  (API с фолбэком на HTML-парсинг), распаковывает tar.gz/zip, копирует в engine/arm64-v8a/.
  ВАЖНО: репозиторий сменился ggerganov → ggml-org; формат архива сменился
  *-android-aarch64.zip → *-android-arm64.tar.gz (с ~b9500+).
- `_patch_pyproject.py` — Python: сканирует engine/arm64-v8a/*.so, автоматически
  генерирует/обновляет секцию [tool.flet.android.libs] в pyproject.toml.
- `setup_engine.bat` — вызывает оба скрипта (единая точка входа для пользователя).
- `update_engine.bat` — бэкап текущего движка + setup_engine.bat + подсказка пересобрать APK.

ИНТЕГРАЦИЯ в build_apk.bat: перед сборкой проверяет наличие engine/arm64-v8a/libllama-server.so
и секции [tool.flet.android.libs] в pyproject.toml. Если движок есть, но не прописан —
вызывает _patch_pyproject.py автоматически.

## Оптимизация для Android 15/16 и исправление сборщика библиотек (10.06.2026)
ПРОБЛЕМА: Несмотря на прописывание файлов `.so` в `[tool.flet.android.libs]` в `pyproject.toml`, библиотеки движка `llama-server` фактически НЕ упаковывались в APK (их не было в скомпилированном пакете). Это приводило к ошибке «движок не найден» при запуске локального инференса. 

РЕШЕНИЕ:
1. **Патч сборщика Flet CLI:**
   Выяснено, что официальный CLI сборщика `flet` (версия `0.80.5`) не считывает и не копирует нативные библиотеки из `[tool.flet.android.libs]`. 
   Мы отредактировали локальный скрипт сборщика `site-packages/flet_cli/commands/build_base.py` в Python-окружении, добавив копирование файлов, указанных в этой секции, в папку `android/app/src/main/jniLibs/<abi>/` сгенерированного Flutter-проекта непосредственно перед сборкой Gradle.
2. **Проверка выравнивания ELF-файлов (16 KB page-size):**
   Написан скрипт проверки выравнивания для Android 15/16. Все нативные `.so` файлы библиотек и самого `llama-server` проверены и имеют корректное 16 KB выравнивание сегментов PT_LOAD (в то время как библиотеки Flutter используют 64 KB). Это гарантирует, что они успешно загружаются через `dlopen` на новых 64-битных ядрах с размером страницы 16 КБ.
3. **Обновление SDK и таргета:**
   В `pyproject.toml` добавлены настройки `min_sdk_version = 26` и `target_sdk_version = 36` (Android 16). Сборка успешно компилирует проект, подгружая SDK 36.
4. **Обход политики W^X:**
   Для запуска процессов `llama-server` используется Java-класс `ProcessBuilder` (через JNI / `android_bridge.py`), что обходит системные блокировки Android 12+ на прямой вызов `execve` из Python-потока.
РЕЗУЛЬТАТ: Собран оптимизированный APK размером 144.6 МБ, содержащий все 22 библиотеки движка, полностью совместимый с Android 15 & 16.

- **Сбой API-ключа в настройках (битый ключ):**
  * Обнаружено, что в эмуляторе в поле `openrouter_keys` был сохранён битый ключ (с прикреплённой строкой модели на конце).
  * Решение: очистка данных приложения (`pm clear`) и запись верного ключа в публичный `mobile_settings.json` в `Download/CorePilot`, который теперь корректно считывается благодаря выданным вручную через adb правам `MANAGE_EXTERNAL_STORAGE`.

  * Мы можем загружать исправленные python-файлы (`main.py`) прямо туда через `adb push` и делать перезапуск приложения (`am force-stop` -> `am start`). Они не перезаписываются при перезапуске, что позволяет тестировать изменения мгновенно.
