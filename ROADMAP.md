# План роста и развития CorePilot (AI Factory)

Документ описывает направление развития проекта. Приоритеты — от стабилизации текущего к расширению возможностей. Принцип ядра неизменен: **никакого хардкода, максимальная настраиваемость, эффективность на доступном железе** (ориентир — 8 ГБ VRAM / 16 ГБ RAM, Windows, SSD).

---

## Текущее состояние

Проект работоспособен и прошёл цикл закалки по двум внешним аудитам. Три режима (Конвейер, Менеджер, AI Cleaner), каскадная конфигурация 5 ролей (local/cloud на каждую), фоновый Демон с DAG-зависимостями, песочница с подтверждением diff, карантин очистки с откатом. Графический интерфейс на DearPyGui (10 вкладок, без браузера; Chainlit остаётся запасным). Эскалация Оракул-Титан при застревании локальных моделей.

---

## Недавно завершено

- **Безопасность терминала.** `run_terminal_command` переведён на `shell=False` с блокировкой shell-операторов — инъекция через промпт закрыта.
- **Видимость сбоев.** Аудит «тихих» `except`: операционные ошибки (диск/права/АВ) теперь логируются, штатные ветки осознанно оставлены тихими.
- **Здоровье БД.** Единое долгоживущее SQLite-соединение + WAL + потокобезопасность; чистка `daemon_heartbeat` вероятностная, не на каждый тик. `database is locked` устранён (проверено стресс-тестом).
- **JSON от локальных моделей.** Capability-кэш `_NO_RESPONSE_FORMAT`: связка `(backend, model)`, отвергнувшая `response_format`, больше его не получает. `response_format` по умолчанию отключён для локальных (LM Studio/llama.cpp несовместимы); парсер `_extract_json` — постоянный фолбэк.
- **Юнит-тесты.** Каркас `tests/` (37 тестов): path-safety, парсеры, DAG-логика, нормализация ошибок, чекпойнты, diff. Запуск `pytest` или `python tests/run_all.py`.
- **Производительность поллинга.** Кэш карты статусов/доски по mtime папок + `os.scandir` — без лишнего перечитывания файлов вхолостую.
- **Структура UI.** Монолит интерфейса разбит на `ui_common` / `ui_tabs` / `ui_dpg` с сохранением изоляции (UI знает только сервисный слой).
- **Ротация лога Демона.** `RotatingFileHandler` (2 МБ × 1 backup) — лог не растёт бесконечно при работе неделями.
- **Профили в `user_data/`.** Профили настроек вынесены из `auto_tasks/` в отдельную папку — Демон их больше не принимает за задачи. Атомарная запись, миграция со старых путей.
- **Отказоустойчивость.** Межпровайдерный fallback в `safe_kickoff`: ротация ключей → резервный провайдер (cerebras→groq→gemini→…) → локальная модель. Защита ночного прогона Демона.
- **Управление памятью.** Встроенный llama.cpp-сервер запускается с `--no-mmap` (дефолт контекста 4096, безопасно для 8 ГБ VRAM). Readiness-probe при выгрузке модели между агентами.
- **Бесплатные провайдеры.** Добавлены Cerebras, SambaNova, HuggingFace, Cohere с ротацией ключей и умными синонимами имён.
- **Генерация изображений (8A).** Диспетчер `generate_image`: Forge / ComfyUI / облако (HF Inference API, FLUX). Настройки в UI, ротация ключей, авто-фолбэк.
- **Мобильный чат-бот (8B).** `cloud_api.py` с round-robin ротацией; `client_storage` для настроек (без PermissionError на Android 11+); crash logger; системный промпт; настраиваемые параметры чата.

## Этап 1. Стабилизация и доверие (ближайший)

Цель — превратить «работает» в «работает предсказуемо».

- **Smoke-тест запуска при обновлении зависимостей.** Отслеживать ломающие изменения CrewAI/DearPyGui/litellm; зафиксировать совместимые диапазоны.
- **Расширение тестовой сетки.** Добавить тесты на cleaner-сканеры и каскад `build_role_llm` (сейчас покрыты парсеры, пути, DAG, чекпойнты).

## Этап 2. Опыт пользователя

Цель — снизить порог входа и число обращений в поддержку.

- **Менеджер моделей в UI.** Скачивание GGUF и регистрация в Ollama прямо из интерфейса; индикатор «влезет ли модель в VRAM» по размеру файла.
- **Пресет «один клик под моё железо».** Профиль, который сам расставляет роли под 8 ГБ (одна модель на все локальные роли либо разумный микс) — как стартовая точка, остающаяся полностью редактируемой.
- **Прозрачность выполнения.** Живой прогресс этапов конвейера, оценка времени, токены/стоимость для облачных вызовов.
- **Локализация.** Интерфейс и документация — вынести строки в ресурсные файлы (сейчас тексты вшиты), подготовить EN-версию.

## Этап 3. Возможности агентов

Цель — повысить качество результата и автономность.

- **Память проекта.** Постоянный индекс кодовой базы (краткие саммари файлов, граф зависимостей) для Сборщика — меньше слепого чтения, точнее контекст. Опциональный RAG поверх локального эмбеддера.
- **Реальное выполнение тестов в цикле.** После патча — прогон тестов в песочнице и возврат результата Аудитору как объективного критерия (а не только сверка плана с патчем).
- **Параллелизм с учётом VRAM.** Облачные роли могут идти параллельно; локальные — последовательно с управляемой выгрузкой. Планировщик, понимающий бюджет памяти.
- **Расширяемые инструменты.** Плагинная регистрация инструментов агентов без правки ядра (директория-реестр), чтобы добавлять интеграции декларативно.

## Этап 4. Платформа и масштаб

Цель — выйти за рамки одной машины, сохранив локальность как опцию.

- **Несколько воркеров Демона** с очередью и блокировками (несколько задач параллельно, если позволяет железо/облако).
- **Кроссплатформенность.** Сейчас Cleaner и часть путей — Windows-специфичны (winreg, диски). Выделить платформенный слой, добавить Linux/macOS-реализации сканеров.
- **Удалённый бэкенд моделей.** Возможность указать сервер llama.cpp/Ollama на другой машине в сети (выгрузить тяжёлые модели на отдельный ПК с большой VRAM).
- **Профили окружений.** Базовая версия реализована — именованные профили настроек сохраняются и переключаются во вкладке «Настройки». Дальше: автопереключение по контексту (ноутбук офлайн ↔ десктоп с облаком).

## Этап 5. Надёжность и сообщество

- **Безопасность терминала.** Опциональный белый список команд, ограничение ресурсов выполняемых процессов, аудит-журнал действий.
- **Телеметрия по согласию.** Локальная аналитика успешности задач для самодиагностики (строго офлайн, с явным согласием).
- **Документация для контрибьюторов.** Архитектурный обзор, гайд по добавлению инструментов/бэкендов, шаблоны issue/PR.

## Технический долг (отслеживается отдельно)

- Платформенная привязка Cleaner к Windows (winreg, буквы дисков).
- `litellm.set_verbose` — устаревший способ управления логами, заменить на актуальный API.
- Зависимость выгрузки VRAM в Демоне от срабатывания task-callback конкретной версии CrewAI.
- `response_format: json_object` поддерживается не всеми локальными серверами одинаково — при проблемах с конкретной моделью предусмотрен тумблер отключения (парсер-фолбэк сохранён).

---

## Принципы, которые не меняются

1. **Анти-хардкод** — каждая новая возможность настраиваема (UI/конфиг), значения по умолчанию разумны, но переопределяемы.
2. **Локальность как первый класс** — облако опционально, проект полноценно работает офлайн.
3. **Безопасность по умолчанию** — песочница, карантин, подтверждения; необратимые действия требуют явного согласия.
4. **Эффективность на доступном железе** — решения проверяются на ориентире 8 ГБ VRAM / 16 ГБ RAM, а не только на топовых GPU.
